## 1. Purpose

Make the app usable at narrow window widths. Specifically:

- `AIFeaturesView` collapses gracefully below 1000 px (Phase 3 spec §4.7 gap, flagged in Phase 3 memory as known debt).
- `StatRow` — a shared control used by 34+ module views — reflows below 900 px.
- Phase 4.3 + 4.4 modules (11 new modules) + 5 high-visibility pilot modules get a responsive audit pass with render-smoke tests at both wide and narrow sizes.
- Cross-cutting `INarrowModeService` + binding infrastructure ship once, consumed everywhere.

Non-goals: animations/transitions, mobile/tablet layouts (Vision Doc §12), tablet-specific UX, pixel-perfect visual regression.

## 2. Existing state

- Phase 2 Dashboard already has 900/1000 px responsive handling baked in (see `project_ui_rebuild_phase_2_complete.md`) via local `Grid.ColumnDefinitions` width triggers. No shared service; the knowledge is encoded per-view.
- `AIFeaturesView` layout uses a fixed 2-col Grid — breaks hard below ~1000 px (content overflows).
- `StatRow` is a single `UniformGrid Columns="4"` — at narrow widths the stat tiles squish below readability.
- `MainWindow` top-of-shell row (from Phase 5.2.0 Task 11) is `Auto` height with a banner mounted — responsive by construction, no 5.3 changes needed.
- V2 token namespace clean (Phase 5.1 deleted V1 bridge). Narrow-mode additions use V2 tokens throughout.

## 3. Decisions (defaulted-recommended under AFK mode)

### 3.1 Thresholds — two tiers

**Choice: `IsNarrow = Width < 1000`, `IsVeryNarrow = Width < 900`.**

- `IsNarrow` (triggers at <1000 px): AIFeaturesView compresses (detail sidebar → icon-only 80 px); stat rows reflow from 4 columns to 2.
- `IsVeryNarrow` (triggers at <900 px): stat rows drop to 1 column, module content panels stack vertically, some secondary content collapses.
- Matches Phase 2/3 history (same thresholds already used ad-hoc in Dashboard).

Rejected alternatives:
- Single 960 px threshold — coarser, loses the hybrid "medium" behavior.
- 3-tier with extra `IsCompact` — YAGNI for desktop app; no distinct state for 900–1000 besides what `IsNarrow` already expresses.

### 3.2 `INarrowModeService` architecture

**Choice: Singleton INPC service observing `MainWindow.Bounds`.**

```csharp
namespace AuraCore.Application.Interfaces.Platform;

public interface INarrowModeService : INotifyPropertyChanged
{
    bool IsNarrow { get; }         // Width < 1000
    bool IsVeryNarrow { get; }     // Width < 900
    double CurrentWidth { get; }   // For diagnostic/debug binding

#if DEBUG
    /// <summary>Force IsNarrow/IsVeryNarrow for dev-time testing. DEBUG only.</summary>
    bool? ForceNarrowOverride { get; set; }
#endif
}
```

Implementation: `NarrowModeService` in `AuraCore.Desktop.Services` (parallel to `HelperAvailabilityService`). Ctor takes no deps; MainWindow code-behind subscribes its `Bounds` PropertyChanged to the service's `UpdateWidth(double)` method.

Registered as singleton in `App.axaml.cs` alongside `IHelperAvailabilityService`.

Rejected alternatives:
- Per-window observer — single-window app today; multi-window future is out of scope.
- Static class — untestable.
- Pure value converter — needs data source; service cleaner.

### 3.3 Binding mechanism (hybrid)

**Choice: Direct VM property binding AND a reusable value converter.**

- **Direct binding** (preferred for view-model-driven logic): VMs expose `INarrowModeService NarrowMode { get; }` property. XAML binds `{Binding NarrowMode.IsNarrow}` via DataContext inheritance. Good for conditional rendering, dynamic column counts, layout switches.
- **`BoundsToIsNarrowModeConverter`** (fallback for one-off XAML responsiveness without VM changes): takes `Rect` (from `Window.Bounds`) and threshold param, returns bool. Used where adding a VM property is overkill — e.g., inline `UniformGrid.Columns="{Binding $parent[Window].Bounds, Converter={StaticResource BoundsToIsNarrowModeConverter}, ConverterParameter=1000}"`.

Both mechanisms ship in 5.3.1.

### 3.4 `AIFeaturesView` narrow layout

**Choice: Hybrid collapse at <1000 px; preserve content, compress chrome.**

At `IsNarrow`:
- Main content: 2-col Grid → 1-col stack. Feature cards stack vertically.
- Detail sidebar: compresses from 280 px wide (full text + icons) to 80 px wide (icon-only); icons remain clickable; tooltip shows label.
- Back navigation button (from Phase 3 §4.7): increases tap target, repositioned top-left at narrow.

At `IsVeryNarrow` (<900):
- Sidebar hides entirely; detail access moves into a hamburger-style toggle button.

Rejected alternatives:
- Full sidebar hide at <1000 px — loses navigation affordance too aggressively.
- Overlay drawer — adds Avalonia `SplitView` complexity without clear UX win for desktop.

### 3.5 `StatRow` narrow behavior

**Choice: Reflow via responsive `UniformGrid.Columns` binding.**

`StatRow` is a shared UserControl in `AuraCore.UI.Avalonia/Views/Primitives/StatRow.axaml`. Currently fixed `Columns="4"`. Change to:

```xml
<UniformGrid Columns="{Binding NarrowMode.IsVeryNarrow,
                               Converter={StaticResource BoolToColumnCountConverter},
                               ConverterParameter='1:2'}">
```

`BoolToColumnCountConverter` takes a string `"<true-columns>:<false-columns>"`; at `IsVeryNarrow=true` → 1 column, else 2 columns at `IsNarrow`, else 4 at wide. Since UniformGrid doesn't natively support 3-tier from a single bool, use a `MultiValueConverter`:

```csharp
public sealed class NarrowToColumnCountConverter : IMultiValueConverter
{
    // Inputs: [IsNarrow, IsVeryNarrow]; Output: 1|2|4
    public object Convert(IList<object?> values, ...)
    {
        bool veryNarrow = values.Count > 1 && values[1] is bool v && v;
        bool narrow = values.Count > 0 && values[0] is bool n && n;
        return veryNarrow ? 1 : narrow ? 2 : 4;
    }
}
```

Binding becomes a `MultiBinding`. One converter, used across all 34+ StatRow consumers.

### 3.7 Testing strategy

**Choice: Per-audited-module `WideRender` + `NarrowRender` smoke tests. No pixel comparison.**

Each audited module's view gets TWO `[AvaloniaFact]` tests:

```csharp
[AvaloniaFact]
public void Foo_view_renders_at_wide_size_without_throwing() =>
    AssertRendersAtSize<FooView>(1200, 800);

[AvaloniaFact]
public void Foo_view_renders_at_narrow_size_without_throwing() =>
    AssertRendersAtSize<FooView>(800, 600);
```

`AssertRendersAtSize<TView>(w, h)` helper in `tests/AuraCore.Tests.UI.Avalonia/Helpers/ResponsiveRenderAsserts.cs`:
- Sets `Window.ClientSize = new Size(w, h)`.
- Renders the view inside a host Window.
- Asserts no exceptions, no visible-but-clipped named controls, optionally asserts `Width > 0` on key elements to catch layout collapses.

Rejected:
- Pixel-level visual regression — too brittle; parent spec §6 carried-forward debt.
- Manual-only — breaks CI discipline.

Expected test growth: 16 modules × 2 variants = 32 new tests, plus converter/service unit tests (~10). Target +~42 tests, solution 1893 → ~1935.

### 3.9 Pilot module selection (5 from pre-4.3 set)

**Choice:** Dashboard + SystemHealth + JunkCleaner + RamOptimizer + AIFeaturesView.

Rationale: highest-visibility, representative of different module shapes (dashboard grid, status panel, flagship cleaner, simple tile, feature gallery). If any narrow issues exist in the 5 pilots, they're the most valuable to catch before ship.

### 3.10 Integration with Phase 5.2 banner

**Choice: No changes required.**

`PrivilegeHelperMissingBanner` sits at the top of MainWindow in an `Auto` row. Its inner layout is a horizontal Grid that wraps text automatically and packs the 2 buttons to the right. At narrow widths, text wrapping handles it. No 5.3 changes.

### 3.11 Animation / transitions

**Choice: No transitions; instantaneous layout recalculation.**

Avalonia's native layout pass handles the re-layout on window resize. No `Transitions` defined on affected properties. Reasons:
- Animation-on-resize is distracting and unpolished without careful timing.
- Adds engineering cost and test complexity.
- YAGNI — users don't resize often; if they do, instantaneous is honest.

### 3.12 Dev/test debug flags

**Choice: `#if DEBUG`-gated `INarrowModeService.ForceNarrowOverride` property.**

```csharp
#if DEBUG
public bool? ForceNarrowOverride
{
    get => _forceOverride;
    set
    {
        _forceOverride = value;
        RaisePropertyChanged(nameof(IsNarrow));
        RaisePropertyChanged(nameof(IsVeryNarrow));
    }
}
#endif
```

Values: `null` (default, read actual width), `true` (force `IsNarrow=true`, `IsVeryNarrow=true`), `false` (force wide). Release builds omit the property entirely so it can't be toggled in prod.

Test uses: unit tests flip `ForceNarrowOverride` to verify consumer reaction without resizing. Dev ergonomics: dev menu exposes toggle (future 5.4 debug-panel work, not scoped here).

## 4. Architecture summary

```
INarrowModeService  ──┐
                      ├──  MainWindow.axaml.cs  (subscribes Bounds, calls UpdateWidth)
                      │
                      ├──  VMs  (inject + expose NarrowMode prop for binding)
                      │
                      └──  XAML  (converter + multi-binding paths)

BoundsToIsNarrowModeConverter (value converter, 1 threshold arg)
NarrowToColumnCountConverter (multi-value converter, 1/2/4 col reflow)

DI wiring in App.axaml.cs (alongside IHelperAvailabilityService):
    sc.AddSingleton<INarrowModeService, NarrowModeService>();
```

## 7. Security posture

Low-risk wave — pure UI/layout concerns. No new IPC, no privilege escalation, no network. No new NuGet packages (everything uses existing Avalonia primitives). No new CVE surface.

One micro-concern: the `#if DEBUG`-gated `ForceNarrowOverride` must NOT leak into Release builds. Verified by:
- `#if DEBUG` guard on the property + its backing field.
- A Release-build-only unit test that reflects on `INarrowModeService` and asserts `ForceNarrowOverride` is NOT a property on the interface. (Runs via `#if !DEBUG` in the test project — so it only executes on Release CI.)
