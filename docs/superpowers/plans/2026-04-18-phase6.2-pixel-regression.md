```csharp
using System.IO;
using System.Threading.Tasks;
using Avalonia;
using Avalonia.Controls;
using Avalonia.Headless;
using Avalonia.Media.Imaging;
using Avalonia.Threading;

namespace AuraCore.Tests.UI.Avalonia.PixelRegression;

/// <summary>
/// Render an Avalonia view to an in-memory PNG byte array.
/// Intended for Verify snapshot tests. Initial-render capture — view
/// shows its pre-async-load state (deterministic).
/// </summary>
public static class PixelRegressionHarness
{
    public static async Task<byte[]> RenderViewAsync<TView>(int width, int height)
        where TView : Control, new()
    {
        // Ensure Avalonia UI thread context is active.
        return await Dispatcher.UIThread.InvokeAsync(async () =>
        {
            var view = new TView();

            var window = new Window
            {
                Width = width,
                Height = height,
                SystemDecorations = SystemDecorations.None,
                Content = view,
            };

            // Show so the headless platform performs a render pass.
            window.Show();
            window.Measure(new Size(width, height));
            window.Arrange(new Rect(0, 0, width, height));

            // Give Avalonia one more dispatcher turn to finalize layout.
            await Dispatcher.UIThread.InvokeAsync(() => { }, DispatcherPriority.Render);

            // Avalonia.Headless exposes CaptureRenderedFrame() on Window via extension.
            // If the exact API name differs, substitute:
            //   - GetLastRenderedFrame()
            //   - WriteRenderedFrameToStream(stream)
            //   - etc. (check Avalonia.Headless docs for the current version)
            using var frame = window.CaptureRenderedFrame()
                ?? throw new System.InvalidOperationException("Headless render produced no frame");

            using var ms = new MemoryStream();
            frame.Save(ms);   // WriteableBitmap.Save(Stream) writes PNG on Avalonia

            window.Close();

            return ms.ToArray();
        });
    }
}
```

**Important**: Avalonia's headless render API has evolved across versions. The subagent must verify the exact method names during build. If `CaptureRenderedFrame()` doesn't exist on `Window`:

- Try `window.CaptureRenderedFrame()` as an extension method — `using Avalonia.Headless;`
- Try `AvaloniaHeadless.CaptureRenderedFrame(window)` as a static helper
- Fall back to `RenderTargetBitmap`:
  ```csharp
  var bitmap = new RenderTargetBitmap(new PixelSize(width, height));
  bitmap.Render(view);
  using var ms = new MemoryStream();
  bitmap.Save(ms);
  ```

Document in the commit which API variant actually worked.

```markdown
# Pixel-Regression Goldens

This directory contains `.verified.png` files — the "golden" pixel snapshots
compared against on every test run by the Verify.Xunit + Verify.ImageSharp
framework.

## How to accept new goldens (after intentional UI change)

When a UI change causes pixel-regression tests to fail, the `.received.png`
files show the new rendering. You need to visually confirm the change is
intended, then promote `.received.png` → `.verified.png`.

### Option 1: Verify CLI tool (recommended)

Install once per machine:

    dotnet tool install -g Verify.Tool

Accept all pending changes in this project:

    cd tests/AuraCore.Tests.UI.Avalonia
    verify accept-all

Accept a single test's pending change:

    verify accept DashboardPixelTests.DashboardView_wide

### Option 2: Per-run auto (dev-only, NEVER CI)

Set an environment variable to auto-accept any pixel diff on next run:

    # PowerShell
    $env:VERIFY_AutoVerify = "true"; dotnet test

    # Bash
    VERIFY_AutoVerify=true dotnet test

**Do not commit this mode to CI.** It masks regressions.

### Option 3: Manual file rename

For single-file accepts, rename in file explorer or via shell:

    cd tests/AuraCore.Tests.UI.Avalonia/goldens
    mv DashboardPixelTests.DashboardView_wide.received.png DashboardPixelTests.DashboardView_wide.verified.png

## When to accept vs. when to fix code

**Accept** if the `.received.png` diff matches an intentional change:
- Layout adjustment you just made
- New content, new icon, color update
- Token change you want to propagate

**Fix code** if the `.received.png` shows:
- View missing entire sections
- Colors wildly off
- Layout totally broken
- Random artifacts

## Tolerance tuning

The harness uses:
- **Per-pixel ARGB tolerance:** 5 / 255
- **Percentage threshold:** 0.1% of pixels allowed to differ

If you see flaky failures (same test fails ~1/10 runs on no code change),
consider relaxing per-test tolerance:

    [AvaloniaFact]
    public async Task MyView_wide()
    {
        var png = await PixelRegressionHarness.RenderViewAsync<MyView>(1200, 800);
        await Verifier.Verify(png, "png")
            .ImageSharpHighTolerance();  // or similar Verify.ImageSharp setting
    }

Check Verify.ImageSharp docs for the exact setting name in the installed
version.

## Adding a new view to pixel regression coverage

1. Add a test method to the appropriate file:
   - Core shell surfaces → `CoreShellPixelTests.cs`
   - Module pages → `ModulePixelTests.cs`
   - New test class for non-fitting views
2. Run `dotnet test --filter "FullyQualifiedName~<YourTest>"` → it fails
3. Visually inspect the `.received.png` that appears in this folder
4. Accept it via one of the options above
5. Commit both the test code AND the new `.verified.png`

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Test throws exception instead of pixel diff | View constructor has unsatisfied DI dependency | Add service to `AvaloniaTestApplication.Initialize()` |
| All tests fail with pixel diff after pulling main | Someone updated UI tokens without pushing accepted goldens | Pull again after they push, OR re-run their accept workflow locally |
| `.received.png` is blank / all-black | Avalonia render-capture API mismatch | Check `PixelRegressionHarness.cs` against Avalonia version |
```
