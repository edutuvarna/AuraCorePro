## 3. Locked decisions (AFK recommended defaults)

### Q1 — Scope: one wave OR split?
**Decision:** Split into 4 sub-waves (5.5.1/5.5.2/5.5.3/5.5.4).
**Why recommended:** Clean milestone commits enable per-sub-wave rollback. Matches Phases 5.1–5.4 cadence. 9 items across mixed sizes would be chaos as a flat 9-task plan.

### Q2 — Windows IPC strategy for items 6-8?
**Decision:** Flesh out `AuraCore.PrivilegedService` Named Pipe server (do not pursue UAC-elevated in-process calls).
**Why recommended:**
- Umbrella spec §4.6 already contracts Windows to this path via `IShellCommandService`. Switching to in-process UAC at this stage would fork the cross-platform abstraction — 6 Linux engines (5.2.1) and 5 macOS engines (5.2.2) already flow through `RunPrivilegedAsync`; forcing Windows to a separate path breaks that symmetry.
- Defense-in-depth carries over cleanly: daemon whitelists action IDs; client cannot arbitrary-exec.
- UAC in-process would prompt every privileged operation (bad UX on a monitoring app) and require `WindowsPrincipal` checks / manifest embedding complexity we don't need.
- "Current user accepts the first UAC once on service install, then never again" is the cleanest model — matches what Velotic and other production optimizers do.
**Consequence:** Phase 5.5.1 must also build the service-install mechanism (`sc.exe create` wrapped in an admin bootstrap script or a first-run UAC prompt).

### Q3 — Item 1 JunkCleaner + DiskCleanup consolidation?
**Decision:** Design-first pass. Final recommendation: keep separate with clearer demarcation. No code merge.
**Why recommended:**
- Engines already diverge:
  - **JunkCleaner:** cross-platform, AI-driven, lighter — user temp, browser caches, OS trash.
  - **DiskCleanup:** Windows-only, deep/aggressive — Windows Update cache, DirectX shader cache, memory dumps, empty folders, duplicate files.
- Merge would either lose Windows-depth (collapse down to lowest common denominator) OR pollute cross-platform module with Windows-only scanners.
- User concern "iki ayrı cleaner boş yer kaplıyor" → address via **sidebar subtitle disambiguation + Dashboard presentation clarity**, not code merge.
- Keeps surface area for users who want "quick cross-platform" vs "deep Windows" flows.
**Deliverable:** `docs/design/2026-04-17-cleaner-consolidation-decision.md` with rationale + subtitle wording. No code changes this sub-wave.

### Q4 — Item 4 System Health clarity: repurpose/rewrite/keep?
**Decision:** Keep-as-is + clarity additions only (subtitle + intro card explaining what it is and when to use it).
**Why recommended:**
- System Health is the full-detail drill-down view (all vitals one place). Dashboard = real-time gauges (CPU/RAM/GPU/Disk/Health). They serve different needs.
- Repurpose / rewrite = 3–5 days of scope creep into a sub-wave already carrying 3 medium items. YAGNI.
- Intro card tells users "this is the deep-dive version of the Dashboard gauges". One-time read; no rework needed.

## 4. Sub-wave 5.5.1 — Windows IPC foundation + privileged writes

**Estimated size:** L (largest of the wave, ~8–12 tasks).

### 4.1 Named Pipe server build-out (Task 1)

**File:** `src/Service/AuraCore.PrivilegedService.IPC/PipeServer.cs` (currently a literal empty class).

**Design:**
- Server class hosts a `NamedPipeServerStream` on pipe name `"AuraCorePro"` (constant already exists in `PipeNames.cs`).
- `PipeSecurity` ACL restricts access to `BUILTIN\\Authenticated Users` — no anonymous access. (Service runs as LocalSystem; clients authenticate via their own user identity.)
- Per-connection async loop: `WaitForConnectionAsync` → deserialize `IpcRequest` (JSON) → dispatch to action handler → serialize `IpcResponse` → close.
- Dispatcher inspects `ActionId` against a central whitelist (`ActionWhitelist.Windows` in Ops project — mirrors Linux daemon pattern from Phase 5.2.1).
- Every request is logged with timestamp + action ID + caller SID (for audit).
- Failed / rejected actions return `IpcResponse` with `Success = false` + reason; never throw to client.

**Security tests:**
- Non-whitelisted action ID → rejected with `AuthResult: Denied`.
- Command injection attempt (semicolon / pipe / backtick / UTF tricks in arguments) → rejected at whitelist validator.
- Oversized request (> 64 KB) → rejected without processing body.
- Unauthenticated connection attempt → closed immediately.

### 4.2 IPC contracts expansion (Task 2)

**File:** `src/IPC/AuraCore.IPC.Contracts/IpcMessages.cs`.

**New records:**
```csharp
public sealed record DriverScanRequest(string? DriverClass) : IpcRequest;
public sealed record DriverScanResponse(bool Success, string Message) : IpcResponse;
public sealed record DefenderActionRequest(DefenderAction Action, string? Target) : IpcRequest;
public sealed record DefenderActionResponse(bool Success, string Output, int ExitCode) : IpcResponse;
public sealed record ServiceControlRequest(string ServiceName, ServiceAction Action) : IpcRequest;
public sealed record ServiceControlResponse(bool Success, string Output, int ExitCode) : IpcResponse;

public enum DefenderAction { UpdateSignatures, QuickScan, FullScan, SetRealtimeProtection, AddExclusion, RemoveExclusion, RemoveThreat }
public enum ServiceAction { Start, Stop, Restart, SetStartupAutomatic, SetStartupManual, SetStartupDisabled }
```

All requests inherit `IpcRequest` (base class already has `ActionId`). All responses inherit `IpcResponse` (already has `Success`, `Error`).

### 4.3 Ops project implementations (Task 3)

**Files:**
- `src/Service/AuraCore.PrivilegedService.Ops/ActionWhitelist.Windows.cs` (new — central registry)
- `src/Service/AuraCore.PrivilegedService.Ops/DriverOperations.cs` (new)
- `src/Service/AuraCore.PrivilegedService.Ops/DefenderOperations.cs` (new)
- `src/Service/AuraCore.PrivilegedService.Ops/ServiceOperations.cs` (new)
- `src/Service/AuraCore.PrivilegedService.Ops/RegistryOperations.cs` (already exists; leave as-is)

**Action IDs (whitelist):**
- `driver.scan` — `pnputil /scan-devices`
- `driver.export` — `pnputil /export-driver * <path>` (path validated to be `%PROGRAMDATA%\\AuraCorePro\\DriverBackup\\*`)
- `defender.update-signatures` — `Update-MpSignature`
- `defender.scan-quick` — `Start-MpScan -ScanType QuickScan`
- `defender.scan-full` — `Start-MpScan -ScanType FullScan`
- `defender.set-realtime` — `Set-MpPreference -DisableRealtimeMonitoring $false|$true`
- `defender.add-exclusion` — `Add-MpPreference -ExclusionPath <path>` (path validator)
- `defender.remove-exclusion` — `Remove-MpPreference -ExclusionPath <path>`
- `defender.remove-threat` — `Remove-MpThreat -ThreatID <id>` (numeric ID validator)
- `service.start` — `sc.exe start <name>` (service name regex validator: `^[A-Za-z0-9_\\-\\.]{1,256}$`)
- `service.stop` — `sc.exe stop <name>`
- `service.restart` — composite: stop + wait + start
- `service.set-startup` — `sc.exe config <name> start= auto|demand|disabled`

Each op validates inputs against the whitelist signature before execution. Any argument containing `&`, `|`, `;`, `` ` ``, `$(`, redirection `>`/`<`, newlines, or non-whitelisted chars → rejected pre-exec.

### 4.4 Client-side wiring (Task 4)

**File:** `src/Infrastructure/AuraCore.Infrastructure.PrivilegeIpc/Windows/WindowsShellCommandService.cs` (currently returns `HelperMissing`).

**Replacement:**
- Connect to pipe via `NamedPipeClientStream("AuraCorePro")` with 2s timeout.
- Serialize `PrivilegedCommand` to matching typed `IpcRequest` (map `command.Id` → record type).
- Read response; map back to `ShellResult`.
- Failure modes:
  - Pipe connection refused → `AuthResult.HelperMissing` (service not installed / not running)
  - Whitelist rejection → `AuthResult.Denied`
  - Exec timeout → `AuthResult.Denied` with timeout message
  - Normal exec → `AuthResult.AlreadyAuthorized` (service preauthorized at install)

### 4.5 Service installation (Task 5)

**Deliverable:** `scripts/install-privileged-service.ps1` (admin-only PowerShell installer).

**Flow:**
1. Copy published `AuraCore.PrivilegedService.exe` + dependencies to `%PROGRAMFILES%\\AuraCorePro\\PrivHelper\\`.
2. `sc.exe create AuraCorePrivHelper binPath= "<path>" start= auto DisplayName= "AuraCore Privileged Helper"`
3. `sc.exe description AuraCorePrivHelper "Executes whitelisted privileged commands for AuraCore Pro."`
4. `sc.exe start AuraCorePrivHelper`
5. Verification: try connecting to pipe + `PingRequest`; fail install if no response in 5s.

**Uninstaller:** `scripts/uninstall-privileged-service.ps1` (mirror).

**Client-side bootstrap:** First-run UX detects pipe connection failure, surfaces a one-time "Install privileged helper?" dialog with consent copy + UAC elevation. If user declines, privileged features stay inert with `HelperMissing` banner (existing UX from 5.2).

## 9. Architecture additions

### 9.1 Named Pipe server pattern (mirrors Linux/macOS daemon)

```
AuraCorePro client
    │
    │  (NamedPipeClientStream, JSON-over-pipe)
    ▼
AuraCore.PrivilegedService (LocalSystem)
    │  PipeServer ─ per-connection async loop
    │  ActionWhitelist.Windows ─ validates action id + args
    │  Dispatcher ─ routes to DriverOps / DefenderOps / ServiceOps / RegistryOps
    ▼
Process invocation (pnputil / PowerShell / sc.exe) with validated args only
```

Shared contract with Linux/macOS: `IpcRequest`/`IpcResponse` hierarchy + `ActionWhitelist` pattern + `IShellCommandService.RunPrivilegedAsync` client surface.

### 9.2 QuickAction tile pattern

```
DashboardViewModel
  └─ ObservableCollection<QuickActionTileVM>
       │
       └─ QuickActionTileVM(Id, Label, IconGlyph, Command, AccentToken)
            │
            └─ Command binds to engine method (e.g., BloatwareRemovalEngine.RemoveDefaultsAsync)

QuickActionPresets.Windows → static registry of default tiles
  - "Quick Cleanup" → JunkCleaner
  - "Optimize RAM" → RamOptimizer
  - "Remove Windows Bloat" → BloatwareRemoval (new)
```

Future platforms register their own `QuickActionPresets.Linux` / `.MacOS` (deferred past 5.5 unless user signals).

### 9.3 TreeView VM pattern (for Space Analyzer)

```
DirectoryNodeVM : INotifyPropertyChanged
  - Path : string
  - SizeBytes : long
  - IsDirectory : bool
  - IsExpanded : bool { set → if first time: load children }
  - Children : ObservableCollection<DirectoryNodeVM>  (lazy)
```

## 11. Security checklist (Phase 5.5.1 specific)

- [ ] Pipe ACL: Authenticated Users only (not Everyone, not anonymous).
- [ ] Every action has an entry in `ActionWhitelist.Windows`.
- [ ] Arg validators reject shell metacharacters (`&|;` backtick, `$(`, `<>`, newlines).
- [ ] Path args are canonicalized and validated against a fixed prefix list where applicable (driver backup path, exclusion path).
- [ ] Service runs as LocalSystem (not NetworkService — no network attack surface).
- [ ] Service installer (PowerShell) elevates once at install; first-run client bootstrap requests consent before invoking.
- [ ] Audit log: every request logs timestamp + action ID + caller SID + exit code.
- [ ] Integration tests cover common injection patterns (semicolon, pipe, backtick, unicode homographs).
- [ ] No hardcoded credentials / API keys in Ops project.
