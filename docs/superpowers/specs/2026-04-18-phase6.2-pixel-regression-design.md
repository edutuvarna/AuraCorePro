## 1. Context

Phase 1 (Design System) seeded a known-debt item: "pixel-regression / visual snapshot testing infrastructure gap". Carried forward through Phases 2–5 because feature waves repeatedly prioritized over test-infra. Phase 6 Item 2 closes that debt.

Goal: catch visual regressions — layout shifts, token resolution bugs, accidental style changes — that semantic tests (logical-tree assertions, responsive breakpoint tests) miss. A button moved 3 px, a color drifted, a grid stretched wrong: current suite wouldn't flag these; pixel-diff would.

Scope B (Standard) approved: build the infrastructure + cover ~15 critical views, demonstrate end-to-end workflow, keep coverage expandable. Not every view — YAGNI. If the infrastructure proves reliable and extensive coverage becomes valuable, Item 2 can be re-opened as a micro-phase to reach Scope C (all 50+ views + EN/TR + theme variants).

## 2. Framework stack

- **Verify.Xunit** (snapshot framework for xunit) — MIT, VerifyTests org, mature.
- **Verify.ImageSharp** (PNG snapshot support on top of Verify) — MIT, same org.

Pattern:

```csharp
[Fact]
public async Task DashboardView_wide()
{
    var pngBytes = await PixelRegressionHarness.RenderViewAsync<DashboardView>(1200, 800);
    await Verifier.Verify(pngBytes, extension: "png");
}
```

First run → `DashboardView_wide.received.png` written. Test fails. Developer inspects, renames to `.verified.png` (or uses `verify accept-all` tool). Subsequent runs compare `received` → `verified`, fail if diff exceeds tolerance.

## 3. Golden storage

- Path: `tests/AuraCore.Tests.UI.Avalonia/goldens/<ClassName>.<MethodName>.verified.png`
- Committed to git. PNGs ~20-50 KB each × 30 = ~1 MB total.
- `.received.png` files gitignored via `.gitignore` rule: `*.received.png`
- Every repo operation on goldens is a deliberate accept action — no noise commits.

---

## 4. Tolerance

Configured via Verify.ImageSharp's `ImageSharp.Settings`:

- **Per-pixel ARGB tolerance:** 5 / 255 (absorbs subpixel anti-alias jitter, font-hinting noise).
- **Percentage threshold:** 0.1% of pixels allowed to differ (absorbs minor renderer variation).
- Per-test override available via `[VerifyTolerance(...)]` attribute or inline `SettingsTask.UseImageSharp(...)`.
- Initial settings conservative-but-practical; tune upward if false positives exceed ~1/10 runs.

If the tolerance proves too strict during 6.2.A's proof-of-concept, adjust before landing 6.2.B/C. Don't accumulate flake.

## 5. Rendering pipeline — `PixelRegressionHarness`

Single static helper. Signature:

```csharp
public static class PixelRegressionHarness
{
    public static Task<byte[]> RenderViewAsync<TView>(int width, int height)
        where TView : Avalonia.Controls.Control, new();
}
```

Implementation outline:

1. Ensure Avalonia headless platform initialized (via `AvaloniaTestApplication` from Phase 5 debt sweep).
2. Instantiate `TView`.
3. Create headless `Window` sized `width` × `height`. Set view as content.
4. Measure + arrange + layout pass (Avalonia `MeasureOverride` + `ArrangeOverride`).
5. Render to in-memory bitmap via Avalonia.Headless's render capture (`HeadlessWindowExtensions.CaptureRenderedFrame()` or equivalent — verify exact API during 6.2.A).
6. Encode bitmap as PNG byte[] via SkiaSharp (Avalonia uses Skia under the hood).
7. Return bytes.

**Capture state:** initial render, BEFORE any async data loads complete. This makes tests deterministic (no live disk / network data to drift). Views show their placeholder / empty / pre-scan state. Covers the most-common regression surface (layout + static tokens + initial content).

**DI:** `AvaloniaTestApplication` already bootstraps minimal DI (Phase 5 debt sweep Task B2). Views that call `App.Services.GetService<T>()` during construction resolve against that container; we may need to register additional fake services as new views surface issues.

## 8. Accept-new-goldens workflow

Documented in `tests/AuraCore.Tests.UI.Avalonia/goldens/README.md`:

### Primary: Verify CLI tool

```powershell
dotnet tool install -g Verify.Tool
cd tests/AuraCore.Tests.UI.Avalonia
verify accept-all     # accepts all .received.png → .verified.png
# or: verify accept DashboardView.DashboardView_wide  (single file)
```

### Per-run auto (dev-only, never CI)

```powershell
$env:VERIFY_AutoVerify = "true"
dotnet test
```

Forces test failures to auto-accept. Useful when intentionally mass-updating after a tokens change.

### Manual (file explorer)

Rename `<name>.received.png` → `<name>.verified.png`. Simplest but tedious.

### When to accept

- Only after **visually inspecting** the `.received.png` next to `.verified.png` to confirm the change is intended.
- Layout shifts, new content, color changes = likely intended → accept.
- Entire view corrupted, buttons missing, colors wildly off = likely regression → fix code, don't accept.

## 9. Test categorization

`[Trait("Category", "PixelRegression")]` on every pixel-regression test. Default behavior: runs with main suite (adds ~5-10s wall time; parallelism disabled per Phase 5.4 B1 fix applies uniformly).

Filtering commands:

```powershell
# Only pixel regression:
dotnet test --filter "Category=PixelRegression"

# Everything except pixel regression:
dotnet test --filter "Category!=PixelRegression"
```

Not gated pre-merge — CI doesn't exist yet; when CI wave lands, pixel-regression can be added as a required check.

## 10. Known risks & mitigations

| Risk | Mitigation |
|---|---|
| Font rendering drift across Windows versions | Per-pixel ARGB tolerance absorbs most; escalate to font-embedding if breaks in production |
| Timestamps / "live" data in views | Initial-render capture avoids async-loaded state; if a view paints `DateTime.Now` on mount, add a stub via DI or `TestDateTimeProvider` |
| DI-dependent views crash on render | `AvaloniaTestApplication.Initialize` DI container extended as needed; pattern established Phase 5 debt sweep B2 |
| Avalonia.Headless API drift | Pin Avalonia version; update goldens when upgrading Avalonia (rare — intentional change) |
| PNG non-determinism (metadata, timestamps in file) | Verify.ImageSharp compares pixel data, not file bytes — metadata differences ignored |

## 11. Out of scope

- **Light theme goldens** — Phase 6 Item 3 territory (light theme doesn't exist yet).
- **TR locale goldens** — Phase 6 Item 4 (TR completion sweep); TR strings may differ from EN enough to double the golden count. Revisit there.
- **Mobile/tablet sizes** — Phase 6 Item 5.
- **Pre-merge CI gate** — CI doesn't exist yet; deferred to dedicated CI wave.
- **Animation / GIF regression** — different problem; not addressed.
- **Pixel-diff visualization dashboard** (fancy web UI showing diffs) — nice-to-have; Verify's side-by-side PNG output is sufficient.
- **Golden storage via Git LFS** — 30 PNGs × ~50 KB = 1.5 MB total. In-repo is fine at this scale; LFS evaluation deferred until golden count exceeds 200 or repo complaints surface.
