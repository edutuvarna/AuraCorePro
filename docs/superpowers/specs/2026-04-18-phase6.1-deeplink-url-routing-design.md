## 1. Context

Phase 5.4 shipped `INavigationService` (`src/Core/AuraCore.Application/Interfaces/Platform/INavigationService.cs`) as a cross-cutting navigation contract. Consumers raise `NavigateTo(sectionId)`; the shell (MainWindow) subscribes to `SectionRequested` and routes by ID. Known section IDs today: `ai-recommendations`, `ai-insights`, `ai-schedule`, `disk-health`, plus module IDs registered via the sidebar.

Phase 6 Item 1 adds a Windows URL-scheme handler (`auracore://...`) that translates incoming links into `INavigationService.NavigateTo` calls. Browser/email/chat users click `auracore://disk-health` → AuraCore opens (or focuses) at the Disk Health module. No existing `auracore://` handling exists anywhere in the codebase — built from scratch.

**Scope boundary locked to "Standard" tier** (per user approval): OS scheme registration + URL parsing + single-instance focus; **no** URL query parameters (action-triggering), **no** Linux/macOS scheme registration (Item 5+6 territory), **no** state restore beyond initial navigation.

---

## 2. URL format

`auracore://<section-id>` — flat shape mirroring the existing `INavigationService` ID surface.

**Two allowed shapes:**

| Shape | Example | Routes to |
|---|---|---|
| `auracore://<known-id>` | `auracore://disk-health` | `INavigationService.NavigateTo("disk-health")` |
| `auracore://module/<known-module-id>` | `auracore://module/driver-updater` | `INavigationService.NavigateTo("driver-updater")` (same dispatch; `module/` prefix is namespacing only) |

Anything else (3+ segments, unknown IDs, `%`-encoded traversal, null bytes) → silent reject + debug log; app opens at default Dashboard.

**No query parameters.** `?action=scan-now` style parameterized actions are scoped out of this wave (Scope C future work).

**Known-ID whitelist** (initial; extensible):

- AI sections: `ai-recommendations`, `ai-insights`, `ai-schedule`
- Standalone sections: `disk-health`, `dashboard`, `settings`
- Module IDs: drawn from `ModuleIdsRegistry` (single source of truth — adds support for all ~40 modules automatically as they register)

## 3. Components

### 3.1 `UrlSchemeHandler` (new; `src/UI/AuraCore.UI.Avalonia/Helpers/UrlSchemeHandler.cs`)

Pure logic. Parses an input string, validates against whitelist, returns a `SectionNavigationIntent?`.

```csharp
public enum IntentKind { Section, Module }
public sealed record SectionNavigationIntent(IntentKind Kind, string Id);

public static class UrlSchemeHandler
{
    public static SectionNavigationIntent? Parse(string? input, IReadOnlyCollection<string> knownSectionIds, IReadOnlyCollection<string> knownModuleIds);
    public const string Scheme = "auracore";
}
```

Heavily unit-tested (pure function). No filesystem, no network, no registry. Input/output only.

### 3.2 `UrlGatewayServer` (new; `src/UI/AuraCore.UI.Avalonia/Helpers/UrlGatewayServer.cs`)

Named Pipe server, pipe name `"AuraCorePro.UrlGateway"`. Started in the running AuraCore process. Receives URL payloads from secondary launch attempts, raises `InstanceIntentReceived` event. Single-user ACL (`WindowsIdentity.GetCurrent()` only — narrower than Phase 5.5's Authenticated Users pipe, because deep-links are per-session artifacts).

### 3.3 `UrlGatewayClient` (new; same file or sibling)

Named Pipe client. Used by a secondary AuraCore process to detect existing instance and forward its URL before exiting. Small budget: 500 ms connect timeout. If no existing instance (connect fails), caller falls through to normal startup with the URL as initial-navigation.

### 3.4 `InstanceMutex` helper (new)

Wraps `Mutex("Global\\AuraCorePro.SingletonLock")` or per-user `"Local\\..."` variant. `TryAcquire()` → owner (primary), else secondary. Clean release on process exit.

### 3.5 `Program.cs` / `App.axaml.cs` launch-path wiring

Order of operations on `App` startup:

```
1. Parse args — is there an auracore://... arg?
2. Try acquire InstanceMutex.
3a. Own the mutex:
     - Start UrlGatewayServer (background task).
     - If URL arg present: stash intent for post-MainWindow-loaded dispatch.
     - Normal Avalonia UI startup.
     - MainWindow.Loaded → if stashed intent, NavigateTo.
3b. Did not acquire mutex (another instance running):
     - If URL arg present: UrlGatewayClient.SendAsync(url) → wait for ack → exit(0).
     - If no URL arg: focus the existing main window via Win32 SetForegroundWindow + exit(0).
     - If gateway send fails: fall through to normal startup (defensive — user gets an app, even if second instance).
```

### 3.6 `scripts/install-url-scheme.ps1` + `scripts/uninstall-url-scheme.ps1`

Per-user (HKCU) registration. No admin elevation needed. Writes:

```
HKCU\Software\Classes\auracore
    (Default) = "URL:AuraCore Protocol"
    URL Protocol = ""

HKCU\Software\Classes\auracore\shell\open\command
    (Default) = "<install-path>\AuraCorePro.exe" "%1"
```

Install runs automatically on first AuraCore launch (silent; idempotent — skipped if already registered with correct binary path). Uninstaller removes both keys.

**Auto-install opt-out:** if `HKCU\Software\AuraCorePro\DisableUrlSchemeAutoRegister` DWORD exists and is 1, skip auto-register. For power users who manage registry themselves.

## 4. Security model

### 4.1 Whitelist-first parser

- Unknown IDs → `Parse` returns null. MainWindow logs `[url-scheme] unrecognized URL shape: ..., ignored` at debug level and proceeds to default view.
- No silent display of unknown-URL content to user (no "URL 'javascript:evil' denied" banner that could be abused as attacker messaging surface).

### 4.2 URL = ID lookup key

The URL is never interpreted as a filesystem path, a command, or a script. No process args derived from URL content. No code paths that would allow path traversal, command injection, or argument smuggling.

### 4.3 Pipe ACL

`UrlGatewayServer` pipe is restricted to `WindowsIdentity.GetCurrent()` (current user's SID). Not Authenticated Users. A different user session on the same machine cannot inject URLs into another user's AuraCore instance.

### 4.4 No auto-actions

Deep links only navigate. They never trigger actions (no auto-scan, auto-remove, auto-apply). Navigation surfaces a module page; the user still initiates any privileged or destructive operation by clicking.

### 4.5 No state leakage

Pipe payload is the URL string only. No auth tokens, session IDs, or user identifiers are transmitted via deep links. The URL carries navigation intent — nothing more.

## 5. Defenses against specific attack classes

| Attack | Defense |
|---|---|
| Path traversal (`auracore://../../etc/passwd`, `%2e%2e%2f`, `\..\..`) | Whitelist lookup; traversal segments never in whitelist → reject |
| Command injection (`; rm -rf /`) | URL is parsed, not passed to a shell; no `ProcessStartInfo.Arguments` surface |
| Null byte injection (`auracore://disk-health%00evil`) | URL-decode happens before whitelist check; decoded `disk-health\0evil` not in whitelist |
| Unicode homographs (`auracore://ai%E2%80%A4insights` where dot is one-dot-leader) | String equality via `StringComparison.Ordinal`; Unicode-different bytes → different ID → reject |
| Arg count overflow (`auracore://a/b/c/d/e`) | Only two shapes recognized (1 segment or `module/<id>` = 2 segments); others reject |
| Scheme confusion (`javascript:auracore://...`) | Windows scheme dispatch keys on the leading scheme; `javascript:` hits browser, not AuraCore |
| Cross-session pipe injection | Pipe ACL = current-user SID only |
| URL as attack-feedback channel | No error dialog; attacker cannot learn "hit" vs "miss" from app response |

**Residual risks (in-scope acknowledgments):**

- **User social engineering via legitimate URLs**: attacker sends `auracore://disk-health` expecting user to click. Not a vulnerability — the destination is a valid in-app view.
- **Registry hijacking** (attacker rewrites `HKCU\Software\Classes\auracore` to point at their exe): this is local compromise; out of our defense scope.
- **Scope C future parameterized actions**: when added, they MUST go through an `ArgumentValidator`-style pattern match. Noted for future, not in this wave.

## 6. Testing strategy

### 6.1 Unit tests — `UrlSchemeHandlerTests`

- `Parse("auracore://disk-health", ...)` → `Section, "disk-health"`
- `Parse("auracore://module/driver-updater", ...)` → `Module, "driver-updater"`
- `Parse(null)` → `null`
- `Parse("")` → `null`
- `Parse("auracore://../../system32", ...)` → `null`
- `Parse("auracore://unknown", ...)` → `null`
- `Parse("auracore://ai-insights/extra", ...)` → `null` (3-segment reject)
- `Parse("auracore://module/", ...)` → `null` (missing module ID)
- `Parse("auracore://MODULE/driver-updater", ...)` → `null` (case-sensitive, lowercase literal "module" required)
- `Parse("  auracore://disk-health  ", ...)` → `null` (reject — no whitespace tolerance; whitelisted IDs never contain whitespace, so trimming adds no value and risks masking injection)
- `Parse("auracore:///disk-health", ...)` → triple-slash reject
- 10+ security cases (path traversal, null byte, unicode, overflow)

Target: ~25 test cases covering Parse's full decision tree.

### 6.2 Gateway integration tests — `UrlGatewayTests`

- Spawn `UrlGatewayServer` in-process; connect client; send URL string; verify `InstanceIntentReceived` fires with correct payload.
- Invalid-URL send (unknown ID) → server rejects before raising event (or raises with null intent).
- Oversized payload (>1 KB) → server rejects + closes connection. Client returns failure.
- Pipe ACL check: spawn server; verify ACL allows current user, denies `NT AUTHORITY\ANONYMOUS LOGON`.

Target: 4-6 integration tests, Windows-only trait.

### 6.3 Launch-path smoke test

Harder to automate (requires process spawn). Deferred to manual QA + post-install verify. Noted in plan task.

### 6.4 Install-script dry-run test

`scripts/install-url-scheme.ps1 -WhatIf` outputs intended registry changes without applying. Used in CI (future) to verify no accidental HKLM writes, no admin prompts.

## 8. Rollback

Per-sub-wave milestone commits enable granular revert. Full-wave rollback: `git reset --hard c843f3b` returns to main (Phase 5 debt sweep merge). HKCU registry entries removed via `uninstall-url-scheme.ps1` (no system-wide state).

If scheme registration causes issues on user machines, rollback path: ship a patched AuraCore that runs `uninstall-url-scheme.ps1` on first launch + skip registration. User experience: deep links stop working, but app itself unaffected.
