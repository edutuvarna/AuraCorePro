## Scope

### In scope

1. **Token layer split** — add `<ResourceDictionary x:Key="Light">` sibling in `AuraCoreThemeV2.axaml` with light-mode values for every dark-mode token key.
2. **ThemeService refactor** — migrate from bool `IsDarkMode` to 3-state `AppTheme { Dark, Light, System }`. Keep back-compat properties where feasible to minimize call-site churn.
3. **System-theme detection** — when `AppTheme.System`, resolve to the OS's preferred variant via Avalonia's `Application.Current.ActualThemeVariant` or platform settings. Default to Dark when OS preference is unknown.
4. **Persistence migration** — extend `theme.pref` format from `"dark" | "light"` to `"dark" | "light" | "system"`. Unknown values → dark.
5. **Settings UI** — replace the existing 2-state toggle with a 3-option radio group (Dark / Light / System) using the existing localization keys. Reuse the existing `ThemeLabel` region in `SettingsView.axaml` — no new section; just replace the widget.
6. **Startup behavior** — remove hardcoded `RequestedThemeVariant="Dark"` from `App.axaml`. ThemeService.Initialize() sets the variant based on persisted preference (or System resolve) before `MainWindow` shows.
7. **Pixel gallery Light variant** — add 2 Light-theme variants of the Phase 6.2 `DesignSystemGallery` tests (`Gallery_wide_light`, `Gallery_narrow_light`). Total 4 gallery goldens. New tests apply `ThemeService.SetTheme(Light)` at setup, `ThemeService.SetTheme(Dark)` at teardown to restore.

### Out of scope (deferred)

- **Per-module dark-hardcoded color audit.** Some modules may hardcode dark-only colors (e.g., `Foreground="White"` instead of `{DynamicResource TextPrimaryBrush}`). Deliberate debt sweep is a separate Phase 6.x task — not Phase 6.3.
- **Theme-transition animations.** Instant swap only. Animated transitions can introduce jank at theme-switch time; keeping it out.
- **Follow-OS auto-switch.** When System is selected, we resolve-once at startup. We do NOT re-resolve if the OS theme changes mid-session. User can trigger re-resolve by re-selecting System from Settings.
- **HDR / high-contrast accessibility modes.** Dark + Light covers 95% of the ask; accessibility modes are a separate discussion.
- **Per-primitive pixel-regression expansion.** Gallery Light variant is the extent; individual-control Light snapshots are not added.

## Design decisions (AFK defaults locked)

| # | Decision | Chosen | Why |
|---|---|---|---|
| 1 | Light palette base | Off-white surfaces (`#F8F8FA`), pure white cards (`#FFFFFF`), near-black text (`#0F0F15`) | Modern 2026 convention (Notion, Linear, Vercel). Pure white is harsh; pure black text is over-contrasty. |
| 2 | Default theme for new installs | System | Respects user OS preference. |
| 3 | Default theme when persistence file missing OR unknown | Dark | Current behavior. Keeps existing users' experience unchanged. |
| 4 | Accent brush values across themes | Same hex | Accent = brand identity. Teal/purple/amber/pink are recognizable; changing them for light mode would dilute brand. |
| 5 | Status brush values across themes | Same hex | Already WCAG-compliant on both bg values. |
| 6 | Tinted status surface alphas | Higher in light mode (e.g., 0.10 → 0.18) | Same alpha on white looks flat; bump alpha ~70% for visibility. |
| 7 | Glow shadow alphas | Lower in light mode (e.g., `#33` → `#14`) | Dark glows on light backgrounds look like smudges. Reduce intensity. |
| 8 | Sidebar bg in light | `#F3F3F7` (slightly darker than page bg) | Subtle depth hint without heavy contrast. |
| 9 | Border brushes | Black-alpha in light, white-alpha in dark | Inverting the base color matches the contrast direction. |
| 10 | Theme switcher widget | 3-option radio group | Dropdown ambiguous for 3 states; toggle needs 2 states. Radio = obvious + accessible. |
| 11 | System-theme detection path | Avalonia `Application.Current.PlatformSettings.GetColorValues().ThemeVariant` | Native API; returns `ThemeVariant.Default` if unknown → fall back to Dark. |
| 12 | Mid-session OS-theme tracking | Not implemented | Resolve-at-startup only. Users can manually re-pick System. |
| 13 | ThemeService API shape | `AppTheme { Dark, Light, System }` + `CurrentTheme` prop + `EffectiveVariant` prop | Matches legacy WinUI pattern; `EffectiveVariant` gives the `ThemeVariant` after System is resolved. |
| 14 | Back-compat for callers reading `IsDarkMode` | Keep as computed property from `EffectiveVariant` | Minimize SettingsView churn; it reads display label only. |
| 15 | Pixel tests per-theme strategy | Setup/teardown hook within each Light test method | Avoids global state leak across other tests. Simple `try/finally`. |
| 16 | Sub-wave count | 5 | A (tokens), B (ThemeService), C (Settings UI + App.axaml), D (pixel tests), E (ceremonial close). |

### ThemeService API (after refactor)

```csharp
public static class ThemeService
{
    public enum AppTheme { Dark, Light, System }

    public static AppTheme CurrentTheme { get; private set; } = AppTheme.Dark;
    public static ThemeVariant EffectiveVariant { get; private set; } = ThemeVariant.Dark;

    // Back-compat shim for SettingsView display label.
    public static bool IsDarkMode => EffectiveVariant == ThemeVariant.Dark;

    public static event Action? ThemeChanged;

    public static void Initialize();
    public static void SetTheme(AppTheme theme);
    public static void Toggle();  // Dark <-> Light (skips System)

    // Resolves AppTheme.System to actual ThemeVariant.{Dark|Light}.
    private static ThemeVariant ResolveSystemVariant();
}
```

**Persistence format:**
- `theme.pref` contains one of `"dark"` / `"light"` / `"system"` (lowercase, trimmed).
- Missing file or unrecognized content → `AppTheme.Dark`.
- On write: lowercase string.

### Settings UI

Replace the existing inline toggle at `SettingsView.axaml` with a labeled radio group:

```xml
<StackPanel Spacing="6">
  <TextBlock Text="{Binding Source={x:Static loc:S.Instance}, Path=[set.theme]}"
             FontSize="{DynamicResource FontSizeLabel}"
             Foreground="{DynamicResource TextMutedBrush}"/>
  <TextBlock Text="{Binding Source={x:Static loc:S.Instance}, Path=[settings.chooseTheme]}"
             FontSize="{DynamicResource FontSizeBody}"
             Foreground="{DynamicResource TextSecondaryBrush}"/>
  <StackPanel Orientation="Horizontal" Spacing="14" Margin="0,8,0,0">
    <RadioButton Name="ThemeSystemRb" GroupName="Theme"
                 Content="{Binding Source={x:Static loc:S.Instance}, Path=[settings.systemDefault]}"/>
    <RadioButton Name="ThemeLightRb"  GroupName="Theme"
                 Content="{Binding Source={x:Static loc:S.Instance}, Path=[settings.light]}"/>
    <RadioButton Name="ThemeDarkRb"   GroupName="Theme"
                 Content="{Binding Source={x:Static loc:S.Instance}, Path=[settings.dark]}"/>
  </StackPanel>
</StackPanel>
```

Code-behind (SettingsView.axaml.cs):
- `OnLoaded`: check current `ThemeService.CurrentTheme`, set matching radio's `IsChecked = true`.
- `Checked` handler on each radio: call `ThemeService.SetTheme(AppTheme.*)`.
- Back-compat with existing `ThemeLabel`: remove it (radio group replaces it). Any other reader of the old label uses the same handler path.

## Risk / known concerns

1. **Hardcoded color leaks.** Some module XAML may hardcode `Foreground="White"` or similar. When switching to Light, those become invisible. *Mitigation:* run the app in Light mode manually post-6.3.C, file issues for any offenders, do NOT block Phase 6.3 on fixes — they're Phase 6.x debt.
2. **Pixel-test flake across themes.** Verify.ImageSharp is strict byte-equality. Setting theme at test start may not instantly propagate through `Application.Current.RequestedThemeVariant` — an extra `Dispatcher.UIThread.RunJobs()` after `SetTheme` call in harness may be required. *Mitigation:* if flake arises, harness adapts.
3. **OS-theme detection gap on Linux.** Avalonia on Linux reports `ThemeVariant.Default` (unknown) — falls back to Dark. This is correct behavior (no-op on Linux) but users expecting OS-follow on Linux will be surprised. *Mitigation:* out of scope; document in the Settings tooltip.
4. **Theme variant on tests that don't set it.** The `AvaloniaTestApplication` hard-codes `RequestedThemeVariant=Dark`. After refactor, existing 1611 UI tests still run under Dark by default. Light pixel tests toggle + reset. No risk to baseline.
5. **Glow shadows are scoped via `ThemeDictionaries`.** Current `Glow*` defs are at the outer shared level (line 109-114). They need to move inside `ThemeDictionaries[Dark]` + a new `ThemeDictionaries[Light]` entry with dimmed alphas. This is a 6.3.A concern.
